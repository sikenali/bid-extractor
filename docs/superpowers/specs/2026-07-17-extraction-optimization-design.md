# 招标文件提取优化设计方案

## 概述

针对当前招标文件提取系统中字段取值不精确、页码缺失、正则模式不佳等问题，进行三层系统性优化。

## A. 章节范围限定

### 目标
将规则搜索限定在文档对应章节内，消除跨章节误匹配。

### 实现

#### 1. 章节-分组映射表
在 Go `main.go` 中定义映射：

| 分组 (group) | 匹配章节标题关键词 |
|-------------|-------------------|
| `info` | 项目信息, 项目概况, 招标公告, 投标邀请, 招标条件, 投标须知, 投标人须知, 招标说明, 招标内容, 采购内容, 项目背景 |
| `business` | 商务条款, 投标人资格, 资格要求, 资质要求, 商务要求, 合同条款, 合同主要条款, 付款方式, 交付要求, 售后服务, 质保要求, 联合体, 分包, 转包 |
| `tech` | 技术规格, 技术要求, 技术参数, 技术标准, 技术需求, 采购需求, 货物需求, 服务需求, 技术规格书, 技术规范, 验收标准, 验收要求, 安装调试, 培训 |
| `score` | 评分标准, 评标办法, 评审办法, 评分细则, 综合评分, 评分因素, 评分项, 打分标准, 定标原则, 评标方法, 评分表, 分值分配 |

映射规则：章节标题包含关键词即匹配（`strings.Contains(title, keyword)`），不分大小写。

#### 2. 章节段落分配
- 各章节记录其 `Title` 和 `Content []string`（段落列表）
- 不属于任何标题的段落归入 `info` 组的章节
- 添加一个章节到章节映射的辅助函数

#### 3. 按组限域搜索
- `applyRules` 传入 `groupToParagraphs map[string][]string`
- 对每个规则，根据其 `group` 字段从对应章节段落列表中搜索
- 如果规则组没有对应章节，回退到全文档文本搜索
- 正则搜索也从全文档改为在对应章节文本片段上执行

### 不做的功能
- 不要求精确章节层级匹配，只做关键词包含判断
- 不处理章节嵌套（扁平化匹配）

## B. 智能关键词提取

### 目标
从关键词匹配到的段落中提取结构化值（数字、日期、金额），而非返回整段。

### 实现

#### 1. 限长截断
`extractByKeyword` 找到字段名后：
- 取当前段落字段名后的文本
- 合并后续 1-2 个非空段落（遇到下一个标题或空行停止）
- 总截取字符不超过 200 个中文字符

#### 2. 结构化值优先匹配（按优先级）
在截取的文本范围内依次尝试：

```
优先级 1: 金额格式
  - ¥[\d,]+(?:\.\d{2})?  → "¥100,000"
  - \d+(?:,\d{3})*(?:\.\d{2})?\s*元  → "100000元"
  - 人民币[\u4e00-\u9fa5]+[整元]  → "人民币壹拾万元整"
  - \d+(?:\.\d+)?%\s* | \d+(?:\.\d+)?折  → "30%"

优先级 2: 日期时间
  - \d{4}年\d{1,2}月\d{1,2}日
  - \d{4}-\d{2}-\d{2}
  - \d{1,2}月\d{1,2}日
  - 收到...后\d+个工作[日天]

优先级 3: 分数/数值
  - (\d+(?:\.\d+)?)\s*分
  - 满分\s*(\d+)
  - 分值\s*(\d+)

优先级 4: 简短文本
  - 去除标点开头后取前 80 个字符
  - 不含换行、不含句号分号的截断
```

#### 3. Fallback
- 如果结构化值未匹配到，回退到当前正则模式
- 正则也在章节限定文本上执行

#### 4. 关键字规则新增 `max_length` 字段（选做）
数据库可扩展，但初期在 Go 端硬编码 200 字符限制即可。

## C. 页码近似追踪

### 目标
为每个提取结果提供近似页码，显示为 `P.5` 格式。

### 实现

#### 1. 分页符检测
unioffice 底层使用 `gooxml` 库，可通过 `doc.X().Body` 访问底层 XML。在 `extractDocxWithChapters` 中：

```go
// 遍历段落时检测分页符
for _, para := range doc.Paragraphs() {
    // 检查段落属性中是否含分页符
    runProps := para.X().RPr
    // 或检查段落中是否有 lastRenderedPageBreak
    for _, run := range para.Runs() {
        if hasPageBreak(run) {
            pageNumber++
        }
    }
}
```

具体实现：
- 遍历所有段落时，检查每个 Run 的 `<w:br w:type="page"/>` 元素或 `<w:lastRenderedPageBreak/>`
- 首次遇到分页符时页码从 1 增至 2，以此类推
- 如果文档无分页符，使用段落计数估算：`pageNum = paraIndex / 40 + 1`

#### 2. 页脚页码正则（辅助）
部分文档在页脚有 `第 X 页` 格式。如果分页符检测不足，尝试从页脚文本正则提取。但限于 unioffice 对页脚的访问能力，此方案作为辅助。

#### 3. 段落→页码映射
- 建立 `paraToPage []int` 数组，长度等于段落数
- 每个段落对应一个页码
- 提取时，根据段落索引查找页码

#### 4. 数据流变更
```
Go main.go:
  - extracts 改为 map[string]ExtractResult
    type ExtractResult struct {
      Value string `json:"value"`
      Page  int    `json:"page"`
    }

Node.js docProcessor.ts / upload.ts:
  - 传递 ExtractResult 中的 page

前端 ProjectView.vue:
  - page 字段从硬编码 'P.1' 改为实际页码
  - 显示格式 "P.{page}"（如 page=5 显示 "P.5"）
```

### 不做的功能
- 不追求精确到渲染级页码（docx 本身无此信息）
- 不处理 PDF 页码（PDF 解析暂未实现）

## 数据库规则优化

### 目标
修正现有正则模式，补充缺失字段的规则。

### 修改内容

1. 为以下 UI 字段添加数据库规则（当前缺失）：
   - 评标办法：正则匹配 "评标办法[：:]\s*([^，。\n]+)"
   - 评分表：正则匹配 "评分表" 所在表格段落
   - 评分项：正则匹配 "评分项[：:]\s*([^，。\n]+)"

2. 修正现有模式的边界：
   - 所有 `([^\\n]+)` 改为 `([^。\\n]+)` 避免跨句子捕获
   - 添加 `\s*[：:]?\s*` 统一分隔符处理
   - 评分相关模式限定在含 "分" 或数字的范围内

3. 字段显示增加：
   - 为 business 组补充缺失字段：供货周期, 售后响应, 培训要求
   - 为 score 组补充：评分说明, 评分依据

## 实施顺序

1. Go: 章节检测 + 段落分组（复用现有 chapters 逻辑）
2. Go: 章节-分组映射 + 限域搜索
3. Go: 智能关键词提取（结构化值匹配）
4. Go: 页码追踪（分页符 + 估算）
5. Node.js: 更新类型定义，传递 page/nested extracts
6. 数据库: 补充/修复规则
7. 前端: 更新 sectionData 和 scoreCards 处理真实页码
8. 集成测试

## 风险与注意事项

- 章节检测依赖于文档标题样式（Heading 1-3），无标题文档会回退到全文搜索
- 分页符检测可能不准确（部分 docx 无分页符标记）
- 数据库规则修改后需重新 seed（可用 INSERT OR REPLACE）
- 前端向后兼容：旧 API 返回无 page 数据时显示 "-"

## 验收标准

1. 项目信息中 90% 以上的字段值准确对应
2. 评分卡片显示的数字来自评分表格而非方法论段落
3. 每个提取结果显示真实的页码（误差 ±3 页以内）
4. 商务条款、技术条款、评分标准各标签页均可正常查看
